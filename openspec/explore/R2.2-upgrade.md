## Open threads from R2.1 that matter for R2.2

- **Explore doc thread (3) — `.remcc/version` schema.** R2.1 shipped with `{source_ref, source_sha, installed_at}`. The explore doc flagged "maybe also: list of file paths managed by remcc (so upgrade knows what to overwrite even if the set changes between versions)." That decision belongs to R2.2's design. Recommendation: don't add it now — `TEMPLATE_PATHS` in `install.sh` is the authoritative list, and upgrade can use it from the *new* tag's `install.sh` (it's running the new one).
- **`installed_at` preservation is currently a no-op.** `write_version_marker` reads the previous `.remcc/version` from the working tree to preserve `installed_at` on re-install, but `create_init_branch` rebuilds the branch from `main` first, so the prior file isn't there. Two consequences for R2.2:
  - For `upgrade`, `installed_at` is **meaningful state** that must survive (it's the date the operator first adopted, not the date they last upgraded). The fix is to read from `origin/main:.remcc/version` (or `origin/remcc-upgrade-<ref>:.remcc/version` on re-run), not the working tree. R2.2 should land that fix.
  - The bug is harmless in R2.1 (same source ref + sha → caller never notices the re-stamp), but trivial to expose with R2.2 by definition.

## Carryover prereq from R2.1 (still required by the workflow)

- `WORKFLOW_PAT` repo secret (workflow can't push under `.github/workflows/` without it). `install.sh init` seeds it via `gh-bootstrap.sh`; `install.sh upgrade` should **not** re-seed (the secret is already there on a target that's adopted), but the version-marker check at the top should refuse to run on a target that doesn't have `.remcc/version` yet.
- `package.json#packageManager: pnpm@<version>` — `verify_prereqs` already enforces this in `install.sh`. R2.2 should call the same prereq-verifier (admin, OpenSpec init'd, `pnpm-lock.yaml`, `packageManager`, etc.).

## Suggested first command

`/opsx:propose` — the explore doc is detailed enough that a one-shot proposal with all artifacts is reasonable. If the proposer wants to revisit the `.remcc/version` list-of-paths question or scope the `installed_at`-preservation fix, switch to `/opsx:new`.

Frame for the proposal:

> R2.2 — `install.sh upgrade`. Periodic refresh of the four template-managed files (workflow, claude settings, openspec config, `.remcc/version`) at a newer remcc ref. Mirrors `install.sh init`'s shape: read `.remcc/version`, resolve ref, shallow-clone remcc at that ref, overwrite templates, bump marker (preserving `installed_at`), open a PR. Skips the prereq-verification + `gh-bootstrap.sh` invocation (one-time work). Refuses to run on targets without `.remcc/version` (steer them to `init`). Branch name: `remcc-upgrade` (single branch reused across upgrade attempts, like `remcc-init` is reused — keeps the "PR already open" idempotency check simple).

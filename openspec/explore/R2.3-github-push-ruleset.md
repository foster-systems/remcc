## Why this is on the roadmap

The `.github/**` push ruleset prevents non-admin actors (the bot)
from modifying anything under `.github/**` at push time. Combined
with the approval ruleset on the default branch, it adds a real
defense-in-depth layer: even if a malicious or buggy agent tries
to widen its own workflow permissions, change triggers, or remove
the apply-step guards in a `change/**` branch, the push is
rejected before it reaches a PR. Operator PR review still catches
the same class of issues, but the push-time block fails earlier
and more obviously.

The feature is **GitHub organization-only** — push rulesets
(`target: push`) are rejected on user-owned repos with `"Source
only org-owned repos can have push rules"`. That's why v0.3.x's
implementation had `push_rulesets_supported()` gating it. v0.4.0
dropped the whole code path because the operator's adopters were
all user-owned at the time and the asymmetry was confusing.

Now that the operator is migrating to an org, the gate disappears
and re-adding the ruleset becomes worth the complexity.

## Reference: the v0.3.0 ruleset body to restore

From `git show v0.3.0:templates/gh-bootstrap.sh` — `want_push_ruleset_json`:

```json
{
  "name": "remcc: block bot edits under .github",
  "target": "push",
  "enforcement": "active",
  "conditions": {
    "ref_name": {
      "include": ["~ALL"],
      "exclude": []
    }
  },
  "rules": [
    {
      "type": "file_path_restriction",
      "parameters": { "restricted_file_paths": [".github/**"] }
    }
  ],
  "bypass_actors": [
    { "actor_id": 5, "actor_type": "RepositoryRole", "bypass_mode": "always" }
  ]
}
```

Notes on the body:
- `~ALL` is correct for push rulesets — push rules are not
  branch-scoped; they apply to any ref the actor tries to push to.
- `actor_id: 5` is the `RepositoryRole` admin (same as the
  approval ruleset). The operator (admin) bypasses; the bot does
  not.
- Whether to keep the legacy name `remcc: block bot edits under
  .github` or pick a new one is an open decision (see below).
- This is the **repo-level** shape (what v0.3.x posted to
  `/repos/<owner>/<repo>/rulesets`). The org-level adaptation is
  different — see next section.

## Org-level shape differs from repo-level (verified 2026-05-15)

When this rule is installed at the **organization level**
(`POST /orgs/<org>/rulesets`) instead of per-repo, two parts of
the body change:

1. **`ref_name` is rejected.** The GitHub API returns
   `HTTP 422 "Target ref_name is not supported for push rulesets"`
   if `conditions.ref_name` is present in an org-level push
   ruleset body. (Push rules at org level apply to all refs by
   definition.) Drop the entire `ref_name` block.
2. **`repository_name` is added.** Use
   `conditions.repository_name.include: ["~ALL"]` (or an explicit
   include list) to scope which repos in the org the rule covers.

Bypass actors, target, and the `file_path_restriction` rule are
unchanged. The verified org-level body (live in `foster-systems`
since 2026-05-15) is:

```json
{
  "name": "remcc: block bot edits under .github",
  "target": "push",
  "enforcement": "active",
  "conditions": {
    "repository_name": { "include": ["~ALL"], "exclude": [] }
  },
  "rules": [
    {
      "type": "file_path_restriction",
      "parameters": { "restricted_file_paths": [".github/**"] }
    }
  ],
  "bypass_actors": [
    { "actor_id": 5, "actor_type": "RepositoryRole", "bypass_mode": "always" }
  ]
}
```

R2.3 implementation note: if R2.3 ships as a repo-level bootstrap
change (most likely shape), the v0.3.0 body above is the correct
starting point. But the proposer should verify against the live
API at implementation time — the v0.3.x code may also stop
accepting `ref_name` for repo-level push rulesets at some point,
mirroring the org-level constraint.

## R2.3 scope re-evaluation (2026-05-15)

Given the `foster-systems` org-level rule, the operator's
**primary** use case (their own adopter repos) no longer needs
R2.3 — every repo in `foster-systems` is already protected. The
remaining scenarios where R2.3 would still add value:

1. **Adopters in third-party organizations.** An operator who
   doesn't own the GitHub organization their adopter repo lives
   in can't install an org-level rule there. A repo-level
   bootstrap step would let them harden the individual repo.
2. **Adopters in personal/user-owned accounts.** Push rulesets
   are entirely unavailable on user-owned repos (API rejects
   `target: push` at the repo level too — confirmed in v0.3.x).
   So R2.3 cannot help this case at all. No change.
3. **Org-owned adopters who don't already have a matching
   org-level rule.** Same shape as scenario 1: the bootstrap
   could install at the repo level, or document the
   org-level recipe.

Three candidate dispositions for R2.3:

- **Close R2.3 and document the org-level recipe instead.**
  Cheapest. `docs/SETUP.md` gets a "Hardening: org-level
  `.github/**` push ruleset" section showing the `gh api` call.
  Repo-level support is not added to the bootstrap.
- **Ship R2.3 as a repo-level opt-in for the scenarios above.**
  Roughly the original plan; renamed/scoped to "for adopters
  outside an org-level-rule context."
- **Defer R2.3 indefinitely.** Org-level rule covers the
  operator's case; let the question come back up if a third
  party asks for it.

The proposer should pick one of these explicitly rather than
inheriting the original "ship a repo-level opt-in" framing.

## Open threads (decide in `/opsx:propose` or `/opsx:new`)

- **Trigger mechanism.** Three candidates:
  1. **Auto-on for org-owned targets, silent skip for user-owned.**
     Mirrors v0.3.x behaviour. Simplest for adopters; surprising
     for org-owned adopters who upgrade from v0.4.0 and suddenly
     get a new ruleset on re-bootstrap.
  2. **Explicit opt-in via env var / CLI flag.** E.g.
     `REMCC_BLOCK_GITHUB_PATH=true bash gh-bootstrap.sh` or
     `gh-bootstrap.sh --block-github-path`. Operator picks. Safer
     for the upgrade-existing-org-adopter case.
  3. **Per-repo variable check.** E.g. `REMCC_BLOCK_GITHUB_PATH`
     repo variable controls reconciliation. Most fiddly; defer.
  Recommendation: **option 2** (explicit opt-in). The whole reason
  the operator is migrating to an org is to *enable* this ruleset,
  so they'll happily set a flag. Keeps the surprise factor at zero.
- **Naming.** Reuse `remcc: block bot edits under .github` or
  pick a new name? Reusing preserves continuity with v0.3.x and
  makes legacy-detection trivial; renaming is an opportunity to
  pick something clearer like `remcc: block non-admin edits under
  .github`. Recommendation: **reuse the legacy name**, no
  rationale strong enough to break continuity.
- **Bypass actors.** Keep admin-only? Or also bypass the App? The
  whole point is the App should NOT bypass — that's what makes
  this a hardening control. Recommendation: **admin-only**, same
  as v0.3.x.
- **Uninstall semantics.** `gh-bootstrap.sh --uninstall` should
  remove the push ruleset by name if it was installed. Open: also
  remove the legacy `remcc: block bot edits under .github` from a
  v0.3.x adopter who happens to run `--uninstall` after upgrading
  past v0.4.0? Recommendation: yes, by name — symmetric with the
  approval-ruleset uninstall.
- **Idempotency snapshot.** When opt-in is engaged, the snapshot
  should include a `# github-path push ruleset` section so
  re-runs visibly converge. When opt-in is off, no section.

## Out of scope reminders

- **Not a migration step for legacy adopters.** v0.3.x adopters
  who already have this ruleset keep it; R2.3 doesn't enumerate or
  reconcile their state. (Same forward-only stance as
  simplify-main-ruleset.) If a v0.3.x org adopter wants to verify,
  they look in the GitHub UI.
- **No path-pattern customisation.** `.github/**` is the only
  pattern. No knob for "block more paths" — keep it focused.
- **No relaxing of admin bypass.** The whole point is the bot
  doesn't bypass; don't add an option to make the App a bypass
  actor.
- **Not coupled to anything outside `gh-bootstrap.sh`.** The
  workflow file and Claude settings are unchanged. Only the
  bootstrap surface grows.
